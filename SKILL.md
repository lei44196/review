## 执行流程

### Step 1：读取配置

读 `D:\all_about_job\面试复盘\config.yml`，获取 resume_path、project_paths、output_dir。

### Step 2：解析基本信息

从文件名或询问用户获取：日期、公司名、岗位、轮次。

**命名规则**：飞书文档主标题统一为 `{日期}面试-{公司}`，如 `2026-06-26面试-明悉科技`

### Step 3：读取转录内容

- 将 `.doc` 文件复制一份改名为 `.html`（ColorOS 的 .doc 实际是 HTML 格式）
- 用 Read 工具读取 `.html` 文件，提取 `<pre>` 标签内的对话内容
- **注意**：ColorOS 转录可能只录到候选人单方声音（全部标记为"讲话人 1"），面试官问题需要从候选人回答中反向推断。这种情况下完整对话以候选人陈述为主、按话题自然分段，而非强行编造面试官提问原文

### Step 4a：简历 + 项目结构轻扫（只扫不读细节）

- 用 Read 工具读简历 PDF，提取：职业背景、技能、项目列表
- **扫描目录结构**：对 `project_paths` 中的每个路径，用 Bash 执行 `ls -R` 获取完整目录树，保存到临时备忘（不读具体模块源码——等知道面试问了什么再定向读取）
- 读取各项目的 README 了解概览

### Step 5：创建输出文件夹

```bash
mkdir -p "/d/all_about_job/面试复盘/{date_str}_面试/"
```

### Step 6：保存原始转录

将 `.doc` 文件重命名为 `.html` 并复制到输出文件夹，保存为 `转录文本.html`（原始录音转写，不做任何修改）

### Step 7：生成完整对话 Markdown

写入输出文件夹下的 **完整对话.md**，格式要求：

- 第一行为 `# {日期} {公司} 面试完整对话`
- 按问答主题分节，每节一个二级标题 `## Q{数字}：{主题概括}`
- 每节开头一句话概括该轮问了什么、候选人怎么答的
- 对话正文标注 **面试官** / **候选人**（加粗）
- 保留完整内容，不删减
- 目的是让人一眼能看到整场面试问了哪些问题，快速定位到感兴趣的位置
- **同时提取面试技术点清单**：在整理完全文后，单独列出一个"面试涉技术点"列表（如：多路召回、Memory 机制、ReAct、NL2SQL…），供 Step 4b 定向读取使用

### Step 4b：根据面试内容定向读取项目源码

拿到 Step 7 提取的"面试涉及技术点"列表后，对照 Step 4a 保存的目录树，只读取与面试技术点相关的源文件：

- **grep 定位**：先用 `grep -rl` 在项目目录中搜索面试提到的关键词，精准定位相关文件
- **定向 Read**：只读命中关键词的源文件（每场面试通常 2~5 个模块）
- **路径校验原则**：Step 8/Step 10 中引用的任何文件路径，必须是 Step 4a 目录树中存在 + 本步骤真实读取过的路径，**禁止凭空推断/捏造路径**

### Step 8：用户确认补强重点 → 生成提纲 → 填写详细版

**Step 8a：提问用户**
在完成 Step 7（完整对话）和 Step 4b（源码定向读取）后，向用户提问：
"这次面试有哪些点你觉得需要着重补强？"

- **如果用户回复了重点** → 按用户列的方向整理提纲（Step 8b）
- **如果用户没回复或说"你自己来"** → 结合转录内容自行判断薄弱环节，生成提纲
- **如果用户事后单独发来提纲文件** → 不再提问，直接按提纲生成详细版（走 Step 8c）

**Step 8b：整理提纲**
根据用户回复，在输出文件夹下创建 **复盘.md**（提纲文件）：
- 按用户列出的重点逐条整理为提纲格式
- 每条只写标题和要点占位，留空待填
- 目的是让用户确认方向无误

**Step 8c：生成详细版**
- 保留提纲文件 `复盘.md` 不动
- 复制一份为 `复盘_详细版.md`
- 逐条填入完整回答，格式：
  - **原回答** → **简历/项目实际内容** → **补强回答**
  - 结合简历指标、**Step 4b 定向读取过**的项目源码实现细节
  - **强制约束**：引用的文件路径必须在 Step 4b 中真实读取过，禁止凭空编造
  - **术语纠错规则**：候选人不标准术语（如"主从架构"→Orchestration、"SKU"→Skill）给出标准表述，语气建设性
- 同时给出针对性改进建议和学习方向

### Step 8d：追加面试总览索引

维护一个总览文件 `/d/all_about_job/面试复盘/面试总览.md`，记录每次面试的摘要索引：

- **文件存在性检查**：`-f` 判断文件是否存在
- **不存在则创建**：写入文件头 `# 面试总览` + 表头 `| 日期 | 公司 | 岗位 | 轮次 | 完整对话 | 回答补强 |`
- **每次追加一行**：用 `>>` 追加新记录行，格式 `| {date} | {company} | {position} | {round} | [完整对话]({dir}/完整对话.md) | [回答补强]({dir}/回答补强.md) |`
- **不覆盖已有内容**：只追加，不修改前面的行

### Step 8e：追加八股汇总

维护 `/d/all_about_job/面试复盘/八股汇总.md`，记录每场面试中涉及的纯理论八股题：

- **筛选规则**：只收录与项目无关的纯理论题（如多线程、索引、HTTP、设计模式等），不收录项目专属问题（如 RRF 融合、记忆系统设计）
- **文件存在性检查**：`-f` 判断文件是否存在
- **不存在则创建**：写入文件头 `# 八股汇总` + 说明文字 `每场面试中被问到的纯理论八股题，按日期追加。` + 分隔线 `---`
- **每次追加一段**，格式：
  ```markdown
  ## {date}
  
  ### {问题1标题}
  
  **问题**：{面试官具体怎么问的}
  
  **要点**：
  - {核心知识点 1}
  - {核心知识点 2}
  - {核心知识点 3}
  
  **拓展**：{关联的其他八股知识点，如问了索引可扩展到聚簇索引 vs 非聚簇索引、回表查询、覆盖索引等}
  
  ---
  
  ### {问题2标题}
  ...
  ```
- **内容要求**：知识点完整，附带关联拓展，确保脱离上下文也能当复习材料用
- **不覆盖已有内容**：只追加，不修改前面的行

### Step 9：写入飞书知识库

在生成本地文件后，将精简版复盘写入飞书知识库"面试记录"（space_id: 7653682991478082519）：

**文档格式**：Markdown，用 `docs +create --doc-format markdown` 创建，再用 `wiki +move` 移入知识库

**文档标题**：`{日期}面试-{公司}`（如 `2026-06-26面试-明悉科技`）

**飞书节点层级规则**：
- 主文档（精简版复盘）放在知识库根目录
- **提纲和详细版都上传飞书**，均作为主文档的**子页面**：
  - `复盘.md` → 子节点标题：`{公司} {轮次} — 复盘提纲`
  - `复盘_详细版.md` → 子节点标题：`{公司} {轮次} — 详细复盘`
- 写入步骤：先创建 doc，再用 `wiki +move` 挂载到主文档节点下
  - 需要指定 `--parent-node-token {主文档的 node_token}`
- 可用 `lark-cli wiki +node-list --space-id <space_id>` 查看当前结构

**文档内容结构**（不含原始对话全文）：

```markdown
# {日期}面试

## 面试摘要

**形式**：{几面} | {时长}  
**公司业务**：{面试官介绍的公司业务}  
**面试官关注点**：{关键考察方向}  
**整体评价**：{1-2 句话总结}

---

## 关键问题 & 补强回答

### Q1：{问题标题}

**原回答问题**：{候选人的原始回答}

**补强回答**：
{优化后的完整回答，结合简历/项目细节}

---

### Q2：{问题标题}
...
```

**写入步骤**：
1. 在 `/c/Users/21376/` 下创建临时 `.md` 文件
2. 执行 `lark-cli docs +create --api-version v2 --doc-format markdown --content @_temp.md`
3. 获取返回的 `document_id`
4. 执行 `lark-cli wiki +move --obj-type docx --obj-token {document_id} --target-space-id "7653682991478082519"`
5. 删除临时文件

### Step 10：输出摘要

在对话框中输出：
1. 本地文件路径（转录文本.html、完整对话.md、回答补强.md）
2. 飞书知识库文档链接
3. 回答补强.md 的核心条目概览（哪些问题做了补强）
4. **项目读取审计**：列出 Step 4b 中根据面试技术点定向读取了哪些源文件，并声明回答补强.md 中引用的所有路径均来源于此（如发现有未读取的路径，立即修正）

## 执行约束

1. **纯本地分析** — 不上传录音/简历/项目到外部 API
2. **路径校验** — 读取 config 时校验所有路径有效性
3. **非破坏性原则** — 不修改简历和项目文件
